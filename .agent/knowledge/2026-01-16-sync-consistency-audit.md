# 专家组深度审计：前端全宗切花、组织选择与后端路由的一致性冲突

**审查日期**：2026-01-16
**审查主题**：同步 UI 逻辑与后端映射逻辑的冲突审计

---

### Step 0：审查边界与现状确认

*   **UI 行为**：
    1.  顶部菜单有“全宗切换器”，决定了当前用户的权限范围和视角。
    2.  同步弹窗（如截图）提供了“选择同步组织”的勾选框（展示的是 ERP 账套/组织）。
*   **后端行为**：
    1.  API 接收当前顶部切花的全宗 ID（`currentFonds`）。
    2.  持久化逻辑目前强制使用这个 `currentFonds` 作为所有数据的目的地。

---

### Step 1：法律法规与会计规范对标（合规执行官）

**结论：存在重大合规漏洞。**

*   **问题 1：全宗切花器的误导性。** 如果会计人员在顶部选择了“全宗 A”，但同步弹窗里勾选了属于“全宗 B”的账套。
    *   **当前后果**：数据将全部错存入“全宗 A”。
    *   **合规风险**：这违反了《会计档案管理办法》关于档案独立性的要求。系统不仅“允许”了错误归档，还“诱导”了跨全宗的数据污染。
*   **全宗 vs 组织**：在法律层面，全宗是档案归口的唯一标准。UI 上的“组织选择”若与“全宗”不匹配，这种自由度是极其危险的。

---

### Step 2：系统架构与安全深度审计（架构师）

**结论：逻辑冗余且产生冲突风险。**

*   **必要性审查**：如果后端已经有了稳定的 `accbook_mapping`（账套 -> 全宗），那弹窗里的“选择同步组织”在**路由功能**上是完全冗余的。
*   **架构漏洞**：
    *   **上下文“绑架”**：现在的架构让“同步这个动作”被“当前 UI 视角”绑架了。
    *   **建议方案**：
        1.  **弱化/移除弹窗内的自由勾选**：同步场景应作为“预定义的任务”，其路由逻辑应完全由后端 `accbook_mapping` 驱动。
        2.  **强制路由**：无论用户当前在 UI 顶部切换到哪个全宗，后端处理每一条凭证时，必须以其自身的 `accbookCode` 为准，强制查找并切换到正确的目标全宗上下文。

---

### Step 3：部署、交付与运维评估（交付专家）

**结论：UI 需要极简化，去除干扰项。**

*   **交付痛点**：用户不理解什么是“账套”，什么是“全宗”。给他们太多勾选框反而会增加填错的概率。
*   **优化方案**：
    1.  同步弹窗应显示：“将按照配置同步以下账套：[账套列表]”。
    2.  不再提供勾选，或者勾选仅作为“本次是否采集”的开关，而不决定其去向。
    3.  数据去向应在配置同步场景时（集成设置）就定死，不允许在采集瞬间临时更改路由。

---

### Step 4：专家组联合建议（修订指令）

#### 🛑 致命阻断点
*   **路由优先权冲突**：UI 顶部切花的全宗上下文，**绝对不能**覆盖后端配置的 `accbook_mapping`。

#### 🏛 架构优化建议（针对 implementation_plan.md.routing 的补充）
1.  **[UI 改造]**：在线采集页面的“勾选组织”，应仅作为过滤器传给适配器，后端接收数据后，**必须根据映射关系强行修正 `FondsContext`**，而不是盲目遵循 API 传入的 `currentFonds`。
2.  **[自律校验]**：如果在同步循环中，某个账套在映射表中找不到对应的全宗，系统应抛出异常并停止该条记录的插入，防止产生“无主/错主”档案。

---

**专家联合签置：建议取消采集弹窗中“选择组织”对归口全宗的影响力，由后端映射逻辑全面接管路由。**
