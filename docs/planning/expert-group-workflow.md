# 虚拟专家组审查工作流

> 当提供数据库设计文档 / ER 图 / SQL 脚本 / 核心业务代码 / 部署架构图时，遵循本工作流进行审查。

---

## Step 1：合规红线与架构稳健性审查

### 1.1 合规视角检查（最高优先级）

- [ ] 数据模型是否覆盖 GB/T 39362 要求的关键元数据：
  - 凭证来源、制单人、审核人、记账人、经办人
  - 电子签名信息、时间戳信息
  - 版本号、变更记录、归档状态、保管期限
  - 审计日志（谁在何时对哪些档案做了什么操作）
- [ ] 电子发票、电子凭证原件（XML / OFD）的存储与关联设计
- [ ] 毁损、作废、替换、重归档等业务情形的规则设计

### 1.2 系统架构与安全检查

- [ ] 数据持久化方案：是否支持标准 SQL，能否平滑迁移数据库镜像
- [ ] 存储架构：是否考虑到 PB 级增量数据的长期保存路径
- [ ] 性能瓶颈：评估核心业务逻辑（如签名、加密、大规模导入）的计算开销
- [ ] 安全底座：检查是否有未受控的第三方外部依赖或潜在的安全后门

### 1.3 信息补充

若缺少关键信息：
1. 列出基于现有材料的「合理假设」
2. 指出「需要额外补充的文档或代码片段」

---

## Step 2：安全与架构深度审计

### 2.1 等保与三员管理

- [ ] 是否有清晰的角色与权限模型：
  - 系统管理员
  - 安全保密员
  - 安全审计员
- [ ] 这些角色在系统中是否可被区分、授权和审计？

### 2.2 数据安全与国密集成

- [ ] 存储中的敏感数据是否有加密策略？
- [ ] 传输通道是否预留国密 SSL / VPN 集成点？
- [ ] 是否预留 SM2 / SM3 / SM4 支持点？

### 2.3 防篡改与审计

- [ ] 是否有完备的操作日志与审计日志？
- [ ] 是否有防止「绕过应用的篡改」的设计？
- [ ] 是否有「关键操作需双人确认 / 工作流审批」的机制？

---

## Step 3：部署与运维规划

> 站在「交付 10+ 家大中型企业客户」的角度

### 3.1 自动化部署与安装包设计

- [ ] 如何在国产 Linux 上实现「一键安装」：
  - 依赖检测（CPU、内存、磁盘、OS 版本、数据库版本）
  - 配置向导（数据库连接、存储路径、License 导入等）
  - 安装过程可回滚、可重试

### 3.2 离线 / 无外网环境支持

- [ ] 首次安装方案
- [ ] 日常升级与补丁更新方案
- [ ] License 更新或续费方案

### 3.3 数据迁移与老系统对接

- [ ] 已有电子档案系统 / 财务系统 / 影像系统的数据迁移方案
- [ ] 数据清洗、口径统一与元数据补齐方案

### 3.4 运维成本控制

- [ ] 自诊断脚本、日志打包工具
- [ ] 监控指标设计（Prometheus / 国产监控系统）

---

## 材料提供顺序（推荐）

1. 数据库设计文档（ER 图 / SQL 脚本）
2. 核心业务代码（归档流程、签名校验、审计日志相关模块）
3. 当前部署架构图与安装说明文档

---

## 审查原则

每一轮审查需：

1. **先做「假设与边界」说明**
2. **给出「当前最严谨可行的诊断与改进建议」**
3. **标注「高优先级需要补充的材料」**
