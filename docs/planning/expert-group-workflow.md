# 虚拟专家组审查工作流

## 执行级检查清单（可直接使用版）

> 本文档定义「虚拟专家组」在实际审查中的 **逐项检查清单与判定口径**。
> 所有结论必须基于**可验证事实**，不得仅凭经验判断。

---

## 使用前置说明（强制）

* 本清单 **必须在《Step 0：假设与边界声明》完成后执行**
* 若发现关键材料缺失，应立即记录为 **🛑 潜在阻断点**
* 所有问题需在报告中标注 **严重级别**

---

## Step 1：合规红线与会计专业性审查（最高优先级）

### 1.1 电子会计档案元数据完整性（红线）

**检查项**

* [ ] 是否覆盖电子会计档案必备元数据：

  * 凭证来源系统
  * 制单人 / 审核人 / 记账人
  * 业务日期 / 记账日期
  * 档案状态（生成 / 归档 / 作废 / 重归档）
  * 保管期限
  * 版本号与变更记录
  * 操作审计日志

**如何检查**

* 查看 ER 图 / 表结构
* 检查是否有明确字段与关联关系

**风险判定**

* ❌ 缺失核心字段 → 🛑 致命（不满足电子会计档案基本要求）
* ⚠️ 字段存在但无约束 / 无来源 → 高风险

---

### 1.2 电子凭证 / 电子发票原件管理

**检查项**

* [ ] 是否保存原始电子凭证（如 XML / OFD）
* [ ] 是否与会计凭证建立**可追溯关联**
* [ ] 是否避免仅保存“图片化结果”

**如何检查**

* 查看存储设计
* 查看凭证与附件关联逻辑

**风险判定**

* ❌ 仅保存图片或转换结果 → 🛑 致命
* ⚠️ 原件存在但无法关联 → 高风险

---

### 1.3 会计专业性合理性检查

**检查项**

* [ ] 借贷是否保持平衡
* [ ] 科目使用是否符合常规会计习惯
* [ ] 红字冲销是否有明确规则
* [ ] 凭证摘要是否具备业务可读性

**如何检查**

* 查看样例数据 / 业务规则
* 对照会计实务常识判断

**风险判定**

* ❌ 违反基本会计逻辑 → 🛑 致命
* ⚠️ 勉强合规但不符合实务 → 中风险

---

## Step 2：系统架构与安全深度审计

---

### 2.1 数据持久化与长期保存能力

**检查项**

* [ ] 是否使用标准 SQL
* [ ] 是否避免数据库厂商专有函数
* [ ] 是否考虑 30 年以上数据迁移与读取

**如何检查**

* 查看 SQL / ORM 使用方式
* 检查是否存在强绑定特性

**风险判定**

* ❌ 强依赖专有特性 → 高风险
* ⚠️ 未考虑长期迁移 → 中风险

---

### 2.2 电子会计档案“四性”技术实现

#### 真实性

* [ ] 是否生成并固化摘要值（Hash）
* [ ] 是否支持签名或验签机制

#### 完整性

* [ ] 是否防止绕过应用直接篡改
* [ ] 是否存在前后手业务单据关联

#### 可用性

* [ ] OFD 是否可独立打开
* [ ] 脱离原系统是否仍可阅读

#### 安全性

* [ ] 是否独立于 ERP 系统存储
* [ ] 是否有访问控制与审计日志

**风险判定**

* ❌ 任一“性”无法成立 → 🛑 致命
* ⚠️ 技术路径存在但未落地 → 高风险

---

### 2.3 权限、审计与三员分立

**检查项**

* [ ] 是否区分：

  * 系统管理员
  * 安全保密员
  * 安全审计员
* [ ] 是否存在权限越权风险
* [ ] 是否所有关键操作都有审计日志

**如何检查**

* 查看权限模型
* 查看日志设计

**风险判定**

* ❌ 无法实现三员分立 → 🛑 致命
* ⚠️ 有角色但不可审计 → 高风险

---

## Step 3：部署、交付与运维可行性审查

---

### 3.1 私有化部署与安装能力

**检查项**

* [ ] 是否支持国产 Linux
* [ ] 是否提供一键安装或自动化脚本
* [ ] 是否具备失败回滚机制

**如何检查**

* 查看安装文档 / 脚本
* 检查依赖检测逻辑

**风险判定**

* ❌ 依赖人工手工部署 → 高风险
* ⚠️ 自动化不完整 → 中风险

---

### 3.2 离线与无外网环境支持

**检查项**

* [ ] 是否支持完全离线安装
* [ ] 升级 / 补丁是否可离线完成
* [ ] License 是否支持离线更新

**风险判定**

* ❌ 必须联网 → 🛑 致命
* ⚠️ 升级流程复杂 → 中风险

---

### 3.3 数据迁移与系统对接

**检查项**

* [ ] 是否有旧系统迁移方案
* [ ] 是否考虑数据清洗与口径统一
* [ ] 是否能补齐缺失元数据

**风险判定**

* ❌ 无迁移路径 → 高风险
* ⚠️ 迁移依赖大量人工 → 中风险

---

### 3.4 运维成本与可维护性

**检查项**

* [ ] 是否提供自检脚本
* [ ] 是否支持日志集中打包
* [ ] 是否有基础监控指标

**风险判定**

* ❌ 运维完全依赖人工 → 高风险
* ⚠️ 监控能力不足 → 中风险

---

## Step 4：结论整理与阻断判断（必须执行）

在完成上述检查后，必须：

1. 汇总所有问题
2. 按严重级别排序：

   * 🛑 致命
   * 高
   * 中
3. 明确：

   * 是否允许进入交付 / 上线阶段
   * 需要补充的关键材料清单

---

## 审查原则（再次强调）

* 不确定 = 标注「待核对」
* 没证据 = 不能下结论
* 缺材料 = 先阻断，不猜测
