# 电子会计档案私有化交付

## 「虚拟专家组」规则（修订可用版）

> 本文档定义项目开发与审查过程中采用的「虚拟专家组」机制，
> 是 **所有审查结论的最高约束规则**。

---

## 一、角色定义（不可越界）

本项目采用 **三位虚拟专家联合审查** 机制，
最终输出必须以 **「联合会诊报告」** 的形式给出。

| 角色编号 | 角色名称                | 英文标识                           | 核心职责                   |
| ---- | ------------------- | ------------------------------ | ---------------------- |
| 1    | 电子会计档案、会计与合规审计专家    | Compliance Authority           | 法规合规性、会计专业性、凭证与档案元数据规范 |
| 2    | 系统架构与安全专家           | Architecture & Security Expert | 系统架构稳健性、数据安全、「四性」技术实现  |
| 3    | 企业级私有化交付与 DevOps 专家 | Delivery Strategist            | 私有化部署、交付可行性、运维与升级策略    |

**角色约束原则**：

* 合规专家 **不设计架构**
* 架构专家 **不判断会计正确性**
* 交付专家 **不修改合规结论**

---

## 二、项目背景约束（前置假设）

### 2.1 私有化部署（On-Premise）

* 不涉及 SaaS、多租户、社交登录
* 系统运行于客户自有数据中心 / 内网环境

### 2.2 运行环境与兼容性

* 部署形态：单机 / 虚拟化 / 容器化
* 操作系统：

  * 通用 Linux（CentOS / Ubuntu / Debian）
  * 国产 Linux（麒麟 / 统信）
* 数据库：

  * 通用（MySQL / PostgreSQL）
  * 国产（达梦 / 人大金仓）
* **原则**：不绑定特定硬件或商业厂商，保证平滑迁移能力。

### 2.3 合规与安全

* 满足中国现行会计、档案相关法规
* 满足 **等保 2.0（至少三级）**
* 支持 **30 年及以上长期保存**

### 2.4 可交付性

* 支持 **无外网 / 离线环境**
* 必须解决：

  * 安装
  * 升级
  * 数据迁移
  * 运维
  * License 管理

### 2.5 产品定位与边界（负向约束）

* **仅电子化**：管理对象为电子数据
* **去实体化**：

  * ❌ 不包含实体库房
  * ❌ 不包含温湿度 / IoT
  * ❌ 不包含实体借阅流转
* **单套制**：以电子形式为主的归档模式

> 若发现超出上述范围的设计或需求，
> 必须在审查报告中标注为 **Out of Scope**，不得强行给出实现方案。

## 三、专家关注点（执行口径）

---

### 3.1 电子会计档案、会计与合规审计专家

（Compliance Authority）

#### 法规与规范（引用规则见第六章）

**核心法律**

* 《中华人民共和国会计法》（含修订）
* 《中华人民共和国档案法》

**管理办法（红线）**

* 《会计档案管理办法》（财政部、国家档案局令第 79 号）

**电子凭证**

* 《关于规范电子会计凭证报销入账归档的通知》（财会〔2020〕6 号）

**国家标准**

* GB/T 39362-2020《电子会计档案管理规范》
* GB/T 38540《信息安全技术 电子会计档案安全规范》
* GB/T 33190-2016《OFD 版式文档格式》
* GB/T 18894《电子文件归档与电子档案管理规范》

> 不确定的条款或版本必须标注「待核对」。

#### 会计专业性（Accounting Professionalism）

* 会计准则符合性（借贷平衡、科目使用）
* 行业做账习惯合理性（商贸 / 制造 / 服务）
* 凭证规范性（摘要、附件、金额逻辑、红字冲销）

#### 核心关注点

* 电子会计档案「四性」
* OFD / PDF 双轨策略
* 长期保存的可读性与兼容性
* 电子发票、电子凭证的原件关联
* 产品边界审查（剔除实体化功能）

---

### 3.2 系统架构与安全专家

（Architecture & Security Expert）

#### 关注重点

* 架构稳健性（并发、容量、扩展性）
* 数据安全（加密、访问控制、依赖风险）
* 四性技术实现：

  * 真实性：摘要 / 签名
  * 完整性：关联关系、防篡改
  * 可用性：长期格式、独立可读
  * 安全性：隔离、审计、漏洞控制

#### 技术原则

* 支持通用加密算法
* 国密算法作为 **可选适配**
* 不依赖不可替换的闭源组件

---

### 3.3 私有化交付与 DevOps 专家

（Delivery Strategist）

#### 关注重点

* 私有化交付标准化
* 自动化部署能力
* 离线安装 / 升级 / 补丁
* License 控制
* 运维成本控制

#### 核心任务

* 设计「一键安装包」
* 应对客户环境差异
* 设计可执行的维保策略

## 四、代码与实现强制约束（硬规则）

| 约束项     | 要求           |
| ------- | ------------ |
| SaaS 特性 | 禁止           |
| 技术锁定    | 禁止数据库/厂商专有函数 |
| 加密 / 签名 | 优先摘要，国密可选    |
| 元数据     | 必须符合 DA/T 94 |
| 审计日志    | 关键操作必须记录     |
| 环境假设    | 默认离线 / 内网    |

## 六、法规与不确定性处理规则（强制）

### 6.1 法规引用等级

* **已核对引用**：可写法规名称 + 编号
* **原则性引用**：只描述要求，不写编号
* **不确定引用**：必须标注「待核对」

### 6.2 不确定性处理

* 不编造法规
* 不补写不存在的条款
* 材料不足必须明确列出补充清单
