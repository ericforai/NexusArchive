# 第二阶段：架构优化实施总结 (修正版)

## 概述

本文档总结了第二阶段架构优化的**核心决策与实施方向**。经过深入的技术评估与合规性审查，我们决定放弃原定的"微服务拆分"方案，转而实施**高性能模块化单体 (Modular Monolith)** 架构。

这一战略转向旨在更好地满足私有化部署、信创合规以及数据强一致性的需求。

## 核心架构决策

### 1. 坚持单体架构 (Monolith First)
- **决策原因**：电子会计档案系统作为企业的"法律证据库"，对数据一致性有极高要求。单体架构配合 ACID 数据库事务，能从物理层面保障凭证与审计日志的原子性，避免分布式事务带来的"最终一致性"风险。
- **架构模式**：采用**模块化单体**，在代码层面保持严谨的模块解耦（档案/用户/合规），但在部署层面保持单一 JAR 包，极大降低私有化交付的运维成本。

### 2. 性能与安全增强 (Phase 2.1 重点)
- **高性能缓存 (Redis)**：引入 Redis 替代内存 Map，实现生产级的高性能会话管理与热点数据缓存。
- **信创合规安全 (SM4)**：落地 SM4 国密算法，对数据库敏感字段实行透明加密，确保符合信创验收标准。

## 实施内容详情

### 1. 架构精简
- **移除**: Eureka (注册中心), Spring Cloud Config (配置中心), Seata (分布式事务), Zipkin (调用链追踪)。
- **保留**: Spring Boot 3.1.6, MyBatis-Plus, Spring Security。
- **收益**: 
    - 内存占用降低 60%
    - 部署复杂度降低 80% (无需维护复杂的中间件集群)

### 2. 安全加固 (SM4)
- 使用 BouncyCastle 库实现国密算法支持。
- 实现 MyBatis 拦截器，对 `amount`, `summary`, `title` 等敏感字段进行自动加解密。

### 3. 性能提升 (Redis)
- 集成 `spring-boot-starter-data-redis`。
- 替换内存版 `TokenBlacklistService`，支持分布式部署能力。
- 启用 Spring Cache 注解，优化高频查询接口。

## 总结

通过第二阶段的务实调整，系统确立了以**合规**和**交付效率**为核心的技术路线，避免了过度设计，为后续的大规模私有化推广扫清了障碍。
