# 专家组联合会诊报告：菜单层级结构（二级 vs 三级）

**审查对象**：电子会计档案系统左侧菜单导航结构
**对比样本**：
- **方案 A（友商）**：三级菜单（会计档案 -> 会计账簿 -> 总账/明细账/日记账...）
- **方案 B（我方）**：二级菜单（档案管理 -> 会计账簿）
**审查日期**：2025-12-16

---

## 一、 审查结论概览

**专家组联合建议：保持方案 B（二级菜单），但需增强"页内导航"。**

| 专家角色 | 投票 | 核心理由 |
| :--- | :--- | :--- |
| **合规专家** | ✅ 方案 B | 这里的"三级"实为元数据分类，非必须导航，GB/T 39362 仅强制要求四类一级分类。 |
| **交付专家** | ✅ 方案 B | **强烈反对方案 A**。将"业务数据"（如现金日记账）硬编码为"系统菜单"会导致交付灾难。 |
| **架构师** | ✅ 方案 B | 菜单应为静态路由，避免因数据变动导致菜单频繁重绘（Flicker）。 |

---

## 二、 详细审查意见

### 1. 电子会计档案与合规审计专家 (Compliance Authority)

**法规对标**：
根据《会计档案管理办法》（财政部、国家档案局令第79号）第六条，会计档案分类为：
1.  **会计凭证** (AC01)
2.  **会计账簿** (AC02)
3.  **财务会计报告** (AC03)
4.  **其他会计资料** (AC04)

**审计观点**：
- **方案 A (友商)** 混淆了"档案分类"与"档案内容"。例如"现金日记账"属于 `ACCOUNT_BOOK_TYPE` 元数据的值，而不是独立的档案门类。
- **方案 B (我方)** 清晰对标国家标准四分类，结构稳固。
- **风险提示**：如果采用方案 A，当未来信创适配导致某些特殊账簿类型变化（如"电子辅助账"）时，需要修改代码维护菜单，增加了合规维护成本。

### 2. 企业级私有化交付与 DevOps 专家 (Delivery Strategist)

**交付痛点分析**：
- **客户差异性**：私有化交付中，90% 的客户并没有全套账簿。有的客户没有"固定资产卡片"，有的没有"纳税申报表"。
- **方案 A 的缺陷**：如果做成三级菜单，对没有某些账簿的客户，菜单会显示大量空白页或报错，或者我们必须为每个客户定制菜单配置（Configuration Hell）。
- **方案 B 的优势**：标准产品只有一套菜单。具体有什么账簿，通过数据查询展示。**极度利于标准化交付和一键安装**。

### 3. 信创与高安全系统架构师 (Xinchuang Architect)

**架构设计原则**：
- **UI/UX 原则**：菜单（Navigation）用于切换功能模块，过滤器（Filter）用于筛选数据。
- **友商设计误区**：友商将"筛选条件"（总账、明细账）做成了"菜单"。这会导致左侧菜单无限膨胀（如果以后有100种报表，菜单要长到无法滚动）。
- **建议**：
    - 保持左侧菜单纯净（仅业务大类）。
    - 在右侧内容区域，设计**"左树右表"**布局。左侧提供一个动态的 Tree View 或 Tab 栏，根据当前全宗实际拥有的账簿类型，动态生成筛选树。

---

## 三、 🏛 架构优化方案（Actionable Advice）

虽然我们拒绝了三级菜单，但友商的"快速定位"体验是值得借鉴的。建议并在**内容区域内部**实现：

| 模块 | 优化建议 | 优先级 |
| :--- | :--- | :--- |
| **页面布局** | **实施"页内分栏"设计**。在"会计账簿"页面内部，左侧增加一个 `200px` 宽度的分类树（Tree Filter）。 | High |
| **动态聚合** | 该分类树应通过 `SELECT DISTINCT book_type FROM arc_account_book WHERE fonds_id = ?` 动态生成。**有数据才显示，无数据不显示**。 | High |
| **元数据驱动** | 确保 `AC02` (会计账簿) 下的子类型（总账、明细账等）严格透传 DA/T 94 标准元数据，用于驱动分类树。 | Medium |

---

## 四、 最终决策

1.  **不修改** `Sidebar` 菜单结构，保持当前的二级分类。
2.  **新建任务**：在档案列表页（如会计账簿页）增加 **"侧边分类栏" (Sidebar Filter)** 组件，实现同等的筛选能力，但保持系统架构的通用性。
