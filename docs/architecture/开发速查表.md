## 前端规则

### 模块导入规则

```typescript
// ✅ 正确 - 使用公开 API
import { useArchiveList } from '@/features/archives';
import { BatchUploadButton } from '@/components/batch';

// ❌ 错误 - 禁止 deep import
import { useArchiveList } from '@/features/archives/hooks/useArchiveList';

// ❌ 错误 - 禁止跨特性内部导入
import { internalHelper } from '@/features/other/internal';
```

### 层级依赖规则

| 从 | 可以导入 | 禁止导入 |
|---|---------|---------|
| `pages/*` | `features/*/index.ts`, `components/*/index.ts`, `layouts`, `auth` | `api`, `store`, `features/*/internal` |
| `components/*` | `components/*/index.ts`, `utils`, `types` | `features`, `pages`, `api`, `store` |
| `features/*` | `features/*/index.ts`, `api`, `utils`, `types` | `features/*/internal` |
| `utils/*` | `utils/*`, `types` | `features`, `pages`, `components`, `api`, `store` |

### 创建新特性

```bash
src/features/<feature-name>/
├── index.ts              # 导出所有公开 API
├── internal/             # 内部实现（禁止外部访问）
├── components/           # 特有组件
├── hooks/               # 特有 hooks
├── api.ts               # API 调用
└── types.ts             # 类型定义
```

### 验证命令

```bash
npm run check:arch        # 检查架构违规
npm run modules:discover  # 列出所有模块
npm run modules:validate  # 验证模块边界
```

## 后端规则

### DDD 模块结构

```
com.nexusarchive.modules.<ModuleName>/
├── api/
│   ├── dto/              # 对外契约 ✅ 可被外部依赖
│   └── controller/       # HTTP 端点 ⚠️  仅限 HTTP 访问
├── app/
│   └── *Facade.java      # 模块入口 ✅ 可被外部依赖
├── domain/               # 核心业务 ❌ 私有
└── infra/                # 技术实现 ❌ 私有
```

### 外部访问规则

```java
// ✅ 正确 - 通过 Facade 访问
import com.nexusarchive.modules.borrowing.app.BorrowingFacade;
import com.nexusarchive.modules.borrowing.api.dto.BorrowingRequest;

// ❌ 错误 - 不能直接访问 Domain
import com.nexusarchive.modules.borrowing.domain.model.Borrowing;

// ❌ 错误 - 不能直接访问 Infra
import com.nexusarchive.modules.borrowing.infra.mapper.BorrowingMapper;

// ❌ 错误 - 不能跨层访问 Mapper
public class MyController {
    @Autowired
    private SomeMapper mapper;  // 禁止！必须通过 Service
}
```

### 创建新模块

```bash
cd nexusarchive-java
./scripts/create-module.sh <ModuleName>
```

### 验证命令

```bash
mvn test -Dtest=ModuleBoundaryTest       # 边界规则
mvn test -Dgroups=architecture           # 所有架构测试
mvn test -Dtest=ComplexityRulesTest      # 复杂度检查
```

## 复杂度限制

| 类型 | 行数限制 | 超限处理 |
|------|----------|----------|
| Service 类 | ≤500 | 提取专用服务类 |
| Entity 类 | ≤400 | 提取内部 DTO/枚举 |
| Controller 类 | ≤600 | 委托逻辑到 Service |

---

## 快速决策

| 问题 | 答案 |
|------|------|
| A 模块需要调用 B 模块？ | 通过 B 的 `app/Facade` 或 `api/dto` |
| 组件需要 API 数据？ | 通过 props 传递或创建专用 hook |
| 共享组件需要状态？ | 使用 props/context，禁止直接用 store |
| utils 需要业务逻辑？ | 移到 features，utils 保持纯净 |
| Service 类太长？ | 提取子服务到 `service/impl/` 子包 |

---

## 违规处理

```
1. 运行检查命令发现违规
2. 根据错误信息定位违规代码
3. 调整导入路径或模块结构
4. 重新验证直到通过
```
