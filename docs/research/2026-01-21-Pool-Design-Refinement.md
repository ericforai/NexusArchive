# 方案重构讨论：易用性与合理性深度分析

## 1. 核心争议点对比

| 维度 | 原方案（统一池+逻辑切换） | 建议方案（分类上传+强校验） |
| :--- | :--- | :--- |
| **易用性** | 稍显复杂：用户在池中需通过筛选看不同门类。 | **极高**：入口即分类，逻辑直觉化。 |
| **合理性** | 基于“待办中心”概念，所有待处理档案聚合。 | 基于“分类操作”概念，符合会计分岗习惯。 |
| **技术难度** | 中：需处理复杂的 `DecidePreviewMode` 逻辑。 | **低**：字段在上传时即锁定，预览只需读字段。 |
| **合规性** | 良好：后期可补齐元数据。 | **优秀**：强制前端校验，数据从出生即合规。 |

---

## 2. 针对用户提议的“分类强校验”深度分析

> **“如果分类选错了，不能上传。”** 

这是一个非常好的逻辑，但需要解决一个核心问题：**系统如何知道用户选错了？**

### 场景 A：智能校验（业界先进做法）
- **实现**：用户上传 PDF，前端/后端调用 **OCR**。
- **逻辑**：如果用户选了“会计账簿（LEDGER）”，但 OCR 识别出这是一张“发票”，系统弹窗：“检测到您选择了账簿，但实际内容为发票，是否更正？”
- **优势**：真正减轻人工压力，实现“分类选对，查看即简单”。

### 场景 B：人工确认（低成本做法）
- **实现**：在上传预览界面，强制用户勾选“我确认此文件属于 [所选门类]”。
- **优势**：技术难度最低，但增加了操作步骤。

## 3. 专家组联合评估

### ⚖️ 电子会计档案与合规专家
- **观点**：支持“入口分类”。DA/T 94 强调元数据在“捕获阶段”就要齐全。如果在上传瞬间就锁定了门类，那么后续的档号生成、保管期限设置将变得极其简单，极大降低了档案整理阶段的复杂性。

### 🏛 系统架构师
- **观点**：如果前端能保证 `archival_category` 的准确性，后端的渲染逻辑确实可以简化为：
  ```javascript
  if (category === 'VOUCHER' && source === 'API') renderCanvas();
  else renderPDF();
  ```
  不需要再通过复杂的嗅探或多重判断。建议保留 **统一的后端表结构**，但提供 **分类的上传/录入入口**。

### 📦 交付与 DevOps 专家
- **观点**：改动难度大幅下降。原本需要修改复杂的“详情预览”通用组件，现在只需要在 `BatchUpload` 增加一个强校验规则即可。

---

## 4. 结论与新思路：重塑“查看按钮”

如果按照你的建议，**“查看按钮就简单了”** 这一点我完全赞同。

**重新修改后的逻辑模型**：
1. **上传即分类**：在`其他档案上传`页面，强制选择门类。
2. **元数据固化**：入库时，`archival_category` 必须有值且不可修改。
3. **按钮逻辑**：
   - 只要是 `VOUCHER` 且来自 API -> 调凭证渲染引擎。
   - 只要是其他（通过上传来的） -> 直接调 PDF/图片预览组件。
   - **完全不再需要“猜测”渲染什么。**

**你怎么看这个折中方案：后端表统一，前端多入口+强引导，查看逻辑极简化？**
