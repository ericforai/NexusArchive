# 企业电签档案元数据及关联关系实操调研报告

## 1. 业界实操：元数据是如何产生的？

在实际的企业应用中，元数据的获取并非全是手工，而是根据企业的数字化程度分为三个层次：

### 层级一：全手工模式 (中小企业/传统企业)
- **现状**：财务人员将档案（PDF/扫描件）上传到档案系统，然后对照内容手工录入“档号”、“会计科目”、“金额”、“年度”等。
- **痛点**：工作量极大，合规性（四性检测）基本靠人工肉眼判断，极易出错。

### 层级二：自动化集成模式 (中大型企业/集成良好的系统)
- **现状**：档案系统与 ERP/费控系统深度集成。
- **操作逻辑**：
    1. **业务端**：员工报销时扫描电子发票，费控系统自动解析发票数据。
    2. **财务端**：一键生成记账凭证，此时 ERP 已经知道“凭证号”和“原始发票 ID”的关系。
    3. **归档端**：归档时，ERP 通过 API 将流转过程中的**结构化数据**直接推送到档案系统。
- **结论**：在这种模式下，**90% 以上的元数据是自动生成的**，无需人工干预。

### 层级三：AI 增强模式 (当前业界最前沿/本项目的目标)
- **现状**：针对那些**没有系统对接**、**来源杂乱**的资料（如通过邮件发送的供应商对账单、扫描的纸质合同）。
- **操作逻辑**：利用 **OCR + NLP** 自动提取摘要，利用**智能算法**自动补全元数据。

## 2. 关联关系元数据 vs. 智能匹配：有了关联还要匹配吗？

这是一个非常敏锐的问题。我们需要厘清两者的本质区别：

### 2.1 逻辑关系：手段与目的
- **元数据（关联标识）是结果**：它是存在数据库表里的一个字段（如 `associated_voucher_id`）。如果这个字段有值，系统就能实现“点击凭证看附件”。
- **智能匹配（Smart Matching）是手段**：它是一个**寻找这个值**的过程。

### 2.2 为什么有了元数据还要做“智能匹配”？

在理想状态下（层级二），ERP 告诉了你关联关系，确实不需要智能匹配。但现实情况往往是：

1. **“数据孤岛”断裂**：通过批量上传进来的 PDF 发票，只有文件，没有任何关联信息。此时，系统里有一堆凭证，也有一堆发票，它们之间没有“线”。
2. **异步同步**：记账凭证是 API 传过来的，但附件是员工之后手动补传的。
3. **元数据不全**：ERP 接口可能只给了凭证号，没给附件 ID。

**结论**：**智能匹配的作用正是为了自动填补“关联关系元数据”这个坑**。
- 如果来源带了关联关系：直接跳过匹配，固化元数据。
- 如果来源没有关联关系：触发智能匹配 → 寻找可能的凭证 → 确认匹配 → **写回元数据关联标识**。

## 3. 企业目前如何做的？ (落地现状)

目前国内大型企业（如中国移动、国家电网等财务共享中心）的方案通常是：

1. **对于常规业务（发票、报销、支付）**：强依赖**报销系统的前端采集**。要求员工在报销时就必须结构化关联，归档时直接取值，这是最稳定的实操。
2. **对于复杂资料（如银行对账单、期末调账资料）**：目前仍然以**半手工**为主。财务人员在“预归档池”里手工勾选关联。
3. **本项目的价值**：我们设计的“预归档库”正是为了解决“半手工”的痛点。
    - **API 来的**：直接入库。
    - **上传来的**：提示“待关联”，提供“智能关联”按钮。
    - **匹配上后**：自动固化关联元数据，从而满足 DA/T 94 的合规审计要求。

**一句话总结：元数据是合规的“门槛”，自动化生成是“目标”，而智能匹配是通往自动化的“桥梁”。**
