## Step 2：系统架构与安全深度审计

### 🛑 2.1 阻断点：Elasticsearch 离线环境适配

**问题描述**：
路线图 Phase 1 核心依赖 Elasticsearch 全文搜索引擎，但当前系统 `ElasticsearchConfig.java` 已被禁用，原因是：

```java
// Disabled due to missing dependencies in offline environment
```

**风险等级**：🔴 高

**影响范围**：
- Phase 1.1 全文搜索引擎 → 完全阻塞
- Phase 1.3 搜索结果增强 → 依赖全文搜索

**架构专家评估**：

| 方案 | 可行性 | 离线适配 | 工作量 |
|------|--------|----------|--------|
| Elasticsearch 嵌入式部署 | 中 | 需打包 JVM + ES | +2周 |
| Apache Lucene 直接集成 | 高 | 纯 Java，可离线 | +1周 |
| PostgreSQL 全文搜索 | 高 | 已有数据库支持 | +3天 |
| 维持现有 LIKE 查询 + 优化 | 高 | 无需额外组件 | +2天 |

> [!IMPORTANT]
> **建议**：鉴于私有化离线部署约束，建议采用 **PostgreSQL 全文搜索 + 现有搜索优化** 作为 MVP 方案，后续视客户需求决定是否引入 Elasticsearch。

### 🟡 2.2 四性检测可视化架构评估

**当前架构**：
- 后端 `FourNatureCheckService` 实现完整检测逻辑
- 前端 `FourNatureReportView.tsx` 以卡片形式展示四项检测结果

**路线图 Phase 2 要求**：
- 雷达图：四性检测评分
- 趋势图：历史合格率
- 饼图：问题分类统计

**架构影响评估**：

| 需求 | 后端支持 | 前端支持 | 缺口 |
|------|----------|----------|------|
| 雷达图（四性评分） | ✅ 已有各项 PASS/FAIL/WARNING 状态 | ❌ 需转换为数值 | 需定义评分规则 |
| 趋势图（历史合格率） | ❌ 无历史检测结果存储 | ❌ 无 | 需新增历史记录表 |
| 饼图（问题分类） | ⚠️ 有单次结果，无聚合统计 | ❌ 需开发 | 需新增聚合 API |

> [!WARNING]
> **发现缺口**：当前系统 **未存储历史检测记录**，无法支持趋势图需求。建议新增 `arc_four_nature_history` 表记录每次检测结果。

### 🟡 2.3 移动端架构评估

**当前架构**：
- `MobileUploadPage.tsx` 仅支持扫码上传
- 无响应式布局框架
- 无 PWA/离线缓存能力

**路线图 Phase 3 要求**：
- 移动端查询
- 移动端审批
- 离线缓存

**架构建议**：

| 方案 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| 响应式 Web | 一套代码多端 | 体验较原生差 | ✅ 推荐 MVP |
| React Native | 接近原生体验 | 需独立维护 | ❌ 不推荐 |
| 微信小程序 | 国内场景适用 | 依赖外网 | ❌ 违反离线约束 |

### 🟢 2.4 审计日志可视化安全评估

**安全考量**：审计日志属于敏感数据，可视化需确保：

- ✅ 只读展示，不修改底层日志 → 路线图符合
- ✅ 权限控制（仅审计员可查看） → 需验证现有权限模型
- ⚠️ 统计数据缓存策略需谨慎，避免数据泄露

## Step 4：结论与联合建议

### 🛑 阻断点汇总

| 编号 | 问题 | 严重级别 | 建议 |
|------|------|----------|------|
| B-1 | Elasticsearch 离线部署未解决 | 🔴 高 | 采用 PostgreSQL 全文搜索作为 MVP |
| B-2 | 四性检测历史记录无存储 | 🟡 中 | 新增历史记录表 |

### 🏛 架构优化方案

| 模块 | 现状 | 优化建议 | 难度 |
|------|------|----------|------|
| 全文搜索 | Elasticsearch 禁用 | PostgreSQL tsvector + gin 索引 | 中 |
| 四性历史 | 无存储 | 新增 `arc_four_nature_history` | 低 |
| 移动端 | 仅扫码上传 | 响应式布局 + PWA | 中 |
| 图表库 | 无 | Ant Design Charts（已引入 Ant Design） | 低 |

### 📦 部署与交付策略

1. **安装包分级**：
   - 标准版：PostgreSQL 全文搜索
   - 增强版：包含 Elasticsearch 完整包

2. **环境检测**：
   - 新增 Elasticsearch 可选检测脚本
   - 根据检测结果自动降级

3. **升级路径**：
   - 标准版 → 增强版需执行索引重建脚本

### ⚖️ 专家联合建议

#### 合规专家建议

> 路线图在合规层面安全，可以推进。建议在 **Phase 2 四性可视化** 中增加「检测规范来源标注」功能，明确展示 DA/T 92-2022 条款依据，提升财务人员对合规性的信任感。

#### 架构专家建议

> 强烈建议将 **Phase 1.1 全文搜索** 拆分为两个阶段：
> 1. **Phase 1.1a**（MVP）：PostgreSQL 全文搜索 + 现有搜索优化
> 2. **Phase 1.1b**（增强）：Elasticsearch 可选集成
> 
> 这样可以快速交付核心价值，同时保留后续扩展能力。

#### 交付专家建议

> 建议在 MVP 阶段优先实施：
> 1. Phase 1.2 多维组合搜索（无需新组件，低风险）
> 2. Phase 2.1 四性仪表盘（Ant Design Charts 成熟可控）
> 
> 这组合能以 **最小交付成本** 覆盖 **最大用户痛点**，符合私有化项目渐进式交付原则。
