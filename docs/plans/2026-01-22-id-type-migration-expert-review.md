> **审查对象**：将 NexusArchive 系统中所有档案相关实体的 ID 类型从 Long 统一迁移到 String  
> **审查日期**：2026-01-22

## Step 0：假设与审查边界声明（强制）

### 已知前提

基于代码分析确认的事实：

1. **ID 策略现状**：
   - **44 个实体**使用 `@TableId(type = IdType.ASSIGN_ID)` — MyBatis-Plus 雪花算法，可生成 String 或 Long
   - **11 个实体**使用 `@TableId(type = IdType.AUTO)` — 数据库自增序列，必须为 Long

2. **核心档案实体已使用 String ID**：
   - `ArcFileContent` (电子文件) — `private String id`
   - `Archive` (档案) — `private String id`
   - `OriginalVoucher` (原始凭证) — `private String id`
   - `BasFonds` (全宗) — `private String id`

3. **使用 Long ID 的实体**（11 个）：
   - `ErpConfig`, `ErpScenario`, `ErpSubInterface` (ERP 配置)
   - `CollectionBatch`, `CollectionBatchFile` (采集批次)
   - `ArchiveBatch`, `ArchiveBatchItem`, `ArchiveSubmitBatch` (归档批次)
   - `ScanWorkspace`, `ScanFolderMonitor` (扫描工作区)
   - `PeriodLock`, `SyncTask`, `SyncHistory`, `IntegrityCheck` (系统运维)

4. **数据库表设计**：
   - 核心表：`acc_archive.id VARCHAR(64)`, `arc_file_content.id VARCHAR(64)`
   - 批次表：`arc_archive_batch.id INTEGER` + `SERIAL` 序列
   - ERP 表：`sys_erp_config.id BIGINT` + `SERIAL` 序列

### 合理假设

- 用户希望统一所有实体 ID 类型以简化代码和外部接口
- 目标是提高跨系统互操作性（与外部 ERP/OA 系统对接）
- 数据量级：单全宗年凭证量 < 100 万条

### 审查边界（Out of Scope）

- 本次评审 **不涉及**：具体迁移脚本编写、代码实现细节
- 本次评审 **不覆盖**：前端代码影响评估、API 兼容性变更

#### 1.2 法规视角分析

> [!IMPORTANT]
> **主键类型不属于电子会计档案合规性要求范围。**

根据 **DA/T 94-2022《电子会计档案管理规范》** 与 **《会计档案管理办法》（财政部79号令）**：
- 法规关注的是 **业务标识符**（档号、凭证号、全宗号）的格式与唯一性
- **技术主键（数据库 ID）** 不属于法规元数据要求
- 系统内部使用 Long 或 String 作为主键，对合规性无直接影响

#### 1.3 会计专业性视角

**无风险**：
- 会计凭证、账簿、报表的业务标识（凭证号、档案号）已为 String 类型
- 借贷平衡、金额精度使用 BigDecimal，与 ID 类型无关

#### 1.4 合规专家建议

> **结论**：从合规角度，此迁移 **无阻断点**，但建议：
> 
> 1. 确保迁移过程中 **业务标识符（档案号、凭证号）不受影响**
> 2. 所有审计日志的 `object_id` 字段需保持与实体 ID 一致
> 3. 迁移后验证 **数据完整性检测功能** 正常运行

## Step 2：系统架构与安全深度审计

### 系统架构与安全专家（Architecture & Security Expert）

#### 2.1 评估结论：**⚠️ 高风险 — 技术复杂度大，收益有限**

| 风险维度 | 风险等级 | 说明 |
|----------|----------|------|
| 数据迁移复杂度 | 🔴 高 | 需修改 60+ 表、关联外键、索引 |
| 代码改动范围 | 🔴 高 | 11 个实体 + 关联 DTO/Repository/Service |
| 外键约束处理 | 🔴 高 | 批次表与文件表存在关联 |
| 性能影响 | 🟡 中 | String 比较略慢于 Long，但可接受 |
| 分布式场景 | 🟢 低 | 当前单体架构，无分布式 ID 需求 |

#### 2.2 技术分析

##### 为何当前设计已经合理？

```text
当前设计策略（混合 ID 类型）是合理的技术选择：

┌─────────────────────────────────────────────────────────────┐
│  业务核心实体（需跨系统互操作）                               │
│  ├── ArcFileContent: String ID (可承载 ERP 原始 ID)          │
│  ├── Archive: String ID                                     │
│  └── OriginalVoucher: String ID                             │
├─────────────────────────────────────────────────────────────┤
│  系统内部实体（不暴露外部接口）                               │
│  ├── ArchiveBatch: Long ID (自增，保证批次顺序)              │
│  ├── ErpConfig: Long ID (配置表，无需跨系统)                 │
│  └── CollectionBatch: Long ID (内部批次管理)                 │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：
- 使用 Long ID 的实体均为 **系统内部管理实体**，不对外暴露 API
- 这些实体使用数据库自增 ID 有明确的技术优势：
  - **批次排序**：`arc_archive_batch.batch_sequence` 需要严格递增
  - **哈希链完整性**：批次 ID 参与防篡改哈希链计算
  - **简化查询**：范围查询 `id > ?` 在 Long 上更高效

##### 迁移可能引入的问题

> [!CAUTION]
> **哈希链防篡改机制可能受影响**

当前 `arc_archive_batch` 表设计：
```sql
-- 批次序列号，用于防止哈希链并发竞态条件
batch_sequence BIGINT NOT NULL
```

如果将批次 ID 改为 String：
1. 需重新设计并发控制机制
2. 哈希链顺序依赖可能被破坏
3. 迁移过程中存在数据一致性风险

#### 2.3 架构专家建议

> **结论**：从架构角度，此迁移 **不推荐执行**。
> 
> **理由**：
> 1. **收益有限**：核心业务实体已使用 String，无额外互操作性收益
> 2. **风险较高**：批次管理、哈希链机制依赖 Long 类型的顺序特性
> 3. **工作量大**：预估 40+ 人日，投入产出比低
> 
> **如确需统一，建议替代方案**：
> - 保持内部 ID 类型不变
> - 在 API 层增加 `externalId` 字段用于外部系统对接

- `arc_archive_batch_item.batch_id` → `arc_archive_batch.id`
- 所有审计日志中的实体引用...

> **每个外键都需要**：
> 1. 临时解除外键约束
> 2. 转换数据类型
> 3. 重建外键约束
> 4. 重建索引
