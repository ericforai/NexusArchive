# 虚拟专家组审查报告：ERP 同步「唯一性冲突」修复方案

**审查日期**：2026-01-16
**审查对象**：ERP 同步 ID 生成逻辑优化（增加全宗后缀）

---

### Step 0：假设与审查边界声明

*   **已知前提**：系统将 `acc_archive.unique_biz_id` 的生成逻辑从 `源系统_凭证号` 修改为 `源系统_凭证号_全宗号`。
*   **合理假设**：
    *   在集团化部署场景下，多个全宗（法人主体）可能共享同一个 ERP 账套或单据流。
    *   `unique_biz_id` 是系统内控制“不重复归档”的核心幂等键。
*   **审查边界**：本轮审查仅关注该 ID 生成逻辑的变更，不涉及底层存储引擎的物理性能分析。

---

### Step 1：法律法规与会计规范对标（合规执行官）

**结论：正确且必要。**

*   **依据**：《会计档案管理办法》（79号令）及 DA/T 94-2022。
*   **分析**：全宗（Fonds）是档案管理的核心物理与逻辑边界。在档案学中，每个全宗必须保持其完整性与独立性。
*   **审计要点**：物理单据（如一张原始发票）在会计实务中可能存在“一张凭证对应多个核算主体”或“单据交叉引用”的情况。若系统限制全局唯一单据号，会导致第二个全宗无法完成合规归档。
*   **合规结论**：将唯一性约束锚定在“单据+全宗”维度，符合“档案按全宗独立管理”的红线要求。

---

### Step 2：系统架构与安全深度审计（架构师）

**结论：方案稳健，但需关注幂等性覆盖。**

*   **架构优势**：
    *   **冲突隔离**：通过复合标识符（Composite ID）消除了多全宗并行同步时的资源竞争。
    *   **四性检查（完整性）**：确保了每一个全宗下的档案实例都有其独立的生命周期追踪 ID，不会因为 ID 冲突导致记录缺失。
*   **🛑 潜在风险（高）**：
    *   **旧数据幂等失效**：存量数据仍使用旧 ID（无后缀），新采集任务会生成带后缀的新 ID。如果对同一张旧单据重新触发同步，系统会因 ID 不匹配而误认为这是“新单据”，从而导致同一个全宗内出现重复记录（一旧一新）。

---

### Step 3：部署、交付与运维评估（交付专家）

**结论：交付友好，平滑迁移。**

*   **部署影响**：该改动仅在应用层逻辑，不涉及 DDL 变更（唯一索引依然生效），对离线交付环境无负面影响。
*   **运维建议**：
    *   **数据清洗指令**：建议为已上线客户提供一个 SQL 脚本，将存量 `unique_biz_id` 批量更新为新格式，以保证幂等逻辑的前后一致性。

---

### Step 4：专家联合建议报告

#### 🛑 阻断点（Showstoppers）
*   **无。** 方案本身不存在违反合规红线或系统崩溃的风险。

#### 🏛 架构优化方案
| 模块 | 现状 | 优化建议 | 难度 |
| :--- | :--- | :--- | :--- |
| **幂等层** | 新旧 ID 格式不统一 | **建议执行存量数据修补**：将历史数据的 `unique_biz_id` 增加当前所属 `fonds_no` 的后缀，确保幂等校验逻辑全局统一。 | 中 |

#### ⚖️ 专家联合建议
*   **合规专家**：确认支持。这是实现“集团化多全宗独立核算”的技术底座。
*   **架构专家**：通过。建议在 `VoucherPersistenceService` 中封装一个 `generateUniqueBizId` 方法，统一调用入口。
*   **交付专家**：通过。请在 `CHANGELOG` 中记录此 ID 变更，以防客户自定义报表依赖此字段。

---

**结论：联合专家组判定该修复方案「合规且架构合理」，建议立即执行。**
