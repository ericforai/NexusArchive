# 穿透联查 - 上游数据缺失问题修复

**日期**: 2026-01-15  
**问题**: 以凭证为中心查询时，左侧上游数据只显示付款单，缺少发票、报销单、申请单  
**原因**: 关系数据中发票、报销单与凭证没有直接关系（需要通过付款单间接关联）  
**状态**: ✅ 已修复（添加直接关系）

---

## 🔍 问题分析

### 问题现象

用户以记账凭证（`JZ-2025-01-001`）为中心查询时：
- ❌ **左侧上游数据**：只显示 1 个付款单（`FK-2025-01-001`）
- ✅ **右侧下游数据**：正常显示报表（`BB-2025-01`）
- ❌ **缺失数据**：4 张发票、报销单、申请单都没有显示

### 根本原因

**原始关系链路**：
```
申请单 → 报销单 → 付款单 → 凭证 → 报表
发票1 ─┐
发票2 ─┼→ 报销单
发票3 ─┤
发票4 ─┘
```

**后端查询逻辑**：
- 只查询**一度关系**（直接关系）：`source_id = '凭证ID' OR target_id = '凭证ID'`
- 找到的直接关系：
  - `付款单 → 凭证`（`FK-2025-01-001` → `JZ-2025-01-001`）
  - `凭证 → 报表`（`JZ-2025-01-001` → `BB-2025-01`）

**结果**：
- ✅ 付款单被识别为上游（因为 `relation.to === centerNode.id`）
- ✅ 报表被识别为下游（因为 `relation.from === centerNode.id`）
- ❌ 发票、报销单、申请单没有被包含（因为是间接关系，需要通过付款单）

---

## ✅ 修复方案

### 数据修复

在 V102 迁移脚本中，添加发票、报销单、申请单与凭证的**直接关系**：

```sql
-- 发票 → 凭证（原始凭证关系）
INSERT INTO acc_archive_relation VALUES ('demo-rel-030', 'demo-reimb-fp-001', 'demo-reimb-jz-001', 'ORIGINAL_VOUCHER', ...);
INSERT INTO acc_archive_relation VALUES ('demo-rel-031', 'demo-reimb-fp-002', 'demo-reimb-jz-001', 'ORIGINAL_VOUCHER', ...);
INSERT INTO acc_archive_relation VALUES ('demo-rel-032', 'demo-reimb-fp-003', 'demo-reimb-jz-001', 'ORIGINAL_VOUCHER', ...);
INSERT INTO acc_archive_relation VALUES ('demo-rel-033', 'demo-reimb-fp-004', 'demo-reimb-jz-001', 'ORIGINAL_VOUCHER', ...);

-- 报销单 → 凭证（业务依据关系）
INSERT INTO acc_archive_relation VALUES ('demo-rel-034', 'demo-reimb-bx-001', 'demo-reimb-jz-001', 'BASIS', ...);

-- 申请单 → 凭证（业务依据关系）
INSERT INTO acc_archive_relation VALUES ('demo-rel-035', 'demo-reimb-sq-001', 'demo-reimb-jz-001', 'BASIS', ...);
```

### 业务逻辑说明

**为什么需要添加这些直接关系？**

1. **符合会计业务逻辑**：
   - 发票是凭证的原始凭证（`ORIGINAL_VOUCHER`）
   - 报销单是凭证的业务依据（`BASIS`）
   - 申请单是凭证的业务起点（`BASIS`）

2. **支持穿透联查**：
   - 以凭证为中心查询时，需要看到完整的业务链路
   - 发票、报销单、申请单都是凭证的**上游数据**
   - 付款单、银行回单、报表是凭证的**下游数据**

3. **数据关系完整性**：
   - 既保留了原有的业务链路关系（申请单→报销单→付款单→凭证）
   - 又建立了直接关系（发票→凭证、报销单→凭证、申请单→凭证）
   - 这样可以同时支持两种查询方式：
     - 以报销单为中心：可以看到发票和付款单
     - 以凭证为中心：可以看到发票、报销单、申请单和付款单
