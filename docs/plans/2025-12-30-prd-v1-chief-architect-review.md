## 🛑 第一章：第一性原理与合规的根本性背离 (The Fundamentals)

### 1.1 "Entity" (法人) vs "Fonds" (全宗) —— 概念的灾难性混淆
PRD 中大量使用 `sys_entity` 和 `entity_id` 来指代档案的核心隔离边界。这是一个典型的**SaaS 商业软件思维**，而非**档案学思维**。

*   **原则**: 根据《档案法》和 DA/T 规范，档案的最高顶层容器是**全宗 (Fonds)**。企业是一个法律实体（Entity），但归档是基于全宗进行的。一个法人可能包含多个全宗，或者随着历史沿革全宗号发生变更。
*   **批判**: 你把 `entity_id` 硬编码为物理隔离键，将来如何处理企业合并、分立时的全宗迁移？如何处理历史全宗？
*   **指令**: 立即重构模型。核心表应使用 `fonds_no` (全宗号) 作为逻辑隔离键，`entity_id` 仅仅是管理维度的属性。**档案是跟着全宗走的，不是跟着法人ID走的。**

### 1.2 "PostgreSQL" 依赖 —— 信创环境的死穴
PRD 第 1.4 节和 3.6 节明确要求 "PostgreSQL 11+" 分区表特性。

*   **风险**: 在信创（Xinchuang）环境下，客户极大概率使用 **达梦 (Dameng)** 或 **人大金仓 (Kingbase)**。虽然它们宣称兼容 PG，但在复杂分区、JSONB索引、存储过程上存在巨大且微妙的差异。
*   **批判**: 你在 PRD 里写死数据库特定特性（Partitioning specific syntax），等于给交付团队挖坑。
*   **指令**:
    1.  **禁止直接依赖 PG 特性**。DB 层必须抽象。
    2.  分区策略必须是**逻辑层面的**（Sharding-JDBC 或 应用层路由），或者在物理层针对不同数据库提供不同的 DDL 实现。
    3.  `JSONB` 在达梦中性能极其不稳定，核心元数据（四性检测所需的 metadata）**必须结构化**，严禁偷懒塞进 JSON。

## ⚠️ 第二章：严谨性的缺失 (Lack of Rigor)

### 2.1 "四性检测" (Four-Natures) —— 形式主义
PRD 3.1 节提到了四性检测，但逻辑极其肤浅。
*   **真实性**: 仅靠 "Hash对比"？**不够**。必须包含数字签名（Digital Signature）的验证，且必须支持国密 SM2/SM3。
*   **可用性**: "扩展名匹配"？**这是儿戏**。我把 `.exe` 改名为 `.pdf` 就能骗过系统？
    *   **要求**: 引入文件头魔法数字（Magic Number）校验，并必须尝试解析文件结构（Dry Parse）。
*   **完整性**: 仅仅检查 "必填字段"？**否定**。必须检查 XML 里的结构化数据与版式文件（OFD/PDF）内的非结构化数据是否**一致**（例如：XML金额 vs PDF OCR金额）。

### 2.2 水印的虚假安全
PRD 2.2 节提到 "前端 Canvas 绘制水印"。
*   **批判**: 任何前端的水印都是**防君子不防小人**。只要我禁用 JS，或者直接调 API 抓包，你的水印就是空气。
*   **指令**: 对于高密级（Secret）档案，**必须**使用服务端合成水印（Server-side Rendering/Baking）。性能可以优化，但安全性不能妥协。

## 🛠 第三章：架构设计的致命伤 (Architectural Flaws)

### 3.1 审计日志的断链风险
PRD 4.2 节提到审计日志，但没有强调**防篡改链 (Tamper-evident Chain)**。
*   **问题**: 如果 DB 管理员 `UPDATE audit_log SET action='View' WHERE action='Delete'`，系统如何发现？
*   **指令**: 每一条审计日志必须包含 `prev_hash`（上一条日志的哈希）。这必须在数据库触发器或应用层强制执行，形成一个 Merkle Chain 结构。

### 3.2 物理归档逻辑的缺失
PRD 把重点放在了电子。但对于会计档案，**很多还是纸质的**。
*   **盲区**: 只有“装盒”，没有“位置管理”（库房-密集架-层-格）。没有“借阅归环”的实物状态流转（在库 -> 借出 -> 待归档 -> 销毁）。
*   **指令**: 补全实物档案管理（Physical Archive Management）的完整生命周期。

## 🔮 第四章：退休老兵的最终建议 (The Verdict)

年轻人，PRD 写得很漂亮，像个互联网产品的 PRD。但这是**档案系统**，是**法律证据系统**，是**企业护城河**。

**我的评分**: 40/100

**接下来的行动**:
1.  **全局替换**: 不再谈 "Entity Isolation"，改谈 "Fonds Isolation" (全宗隔离)。
2.  **去 PG 化**: 数据库设计回退到 SQL-92 标准，特殊特性通过 Adapter 适配。
3.  **重写四性**: 把 "可用性检测" 的颗粒度细化到字节级。

只有做到这些，这个系统才配得上 "集团私有化版" 这几个字。去改吧，别让我失望。
