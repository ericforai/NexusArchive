# 穿透联查“抽丝剥茧”设计方案（付款业务主线）

日期：2026-02-13  
状态：已确认（待实施）

## 1. 目标与范围

### 1.1 目标
用户查看某笔凭证时，系统可自动追溯并展示付款业务全流程，支持“主线自动展开 + 分支手动展开”，并支持批量打印当前可见关联单据。

### 1.2 确认的业务约束
- 主线固定为：`凭证 → 付款单 → 付款申请 → 合同 → 发票 → 回单`
- 若主线中某环缺失：自动跳过继续追溯，并对缺失环节标红告警
- 批量打印范围：`主线节点 + 用户手动展开节点`
- 不与现有系统逻辑冲突：优先复用现有接口、store、三栏视图、抽屉预览

## 2. 交互设计

### 2.1 首屏行为
- 用户输入档号并查询后，系统基于返回关系图自动识别主线。
- 三栏布局保持不变：
  - 左侧：依据/来源（申请、合同、发票等）
  - 中间：核心单据（当前中心节点）
  - 右侧：流向/结果（付款、回单、归档）
- 主线节点高亮显示，非主线节点默认折叠。

### 2.2 缺口展示
- 若主线某步骤缺失，插入前端虚拟“缺口节点”（不落库）。
- 缺口节点展示：缺失类型、建议补录方向、风险提示。
- 链路不中断，继续展示后续可追溯节点。

### 2.3 手动展开
- 点击节点仅展开该节点一层邻接关系，避免信息爆炸。
- 手动展开节点纳入打印范围。

### 2.4 批量打印
- 默认打印主线 + 手动展开分支。
- 打印前执行预检：
  - 预计打印单据数
  - 缺口数量
  - 不可访问附件数量
- 附件去重：同一附件ID仅打印一次。

## 3. 架构与兼容策略（避免冲突）

### 3.1 不改后端主流程
- 继续使用：`GET /relations/{archiveId}/graph`
- 不新增强依赖接口，不改变关系表结构。

### 3.2 前端增量派生
在现有关系图基础上新增“派生状态”：
- `mainPathNodeIds: string[]` 主线节点集合
- `missingMainSteps: Array<{ step: string; expectedType: string }>` 缺口集合
- `manuallyExpandedNodeIds: Set<string>` 用户手动展开集合

### 3.3 关系语义兼容
主线识别优先复用现有关系类型：
- `BASIS`
- `ORIGINAL_VOUCHER`
- `CASH_FLOW`
- `ARCHIVE`

### 3.4 回退策略
- 主线识别失败时：回退到当前全图展示逻辑（不影响现网）
- 附件缺失时：显示“可追溯无影像”，不阻断流程

## 4. 数据流

1. 查询 -> 拉取关系图
2. 前端主线识别器计算主线与缺口
3. UI按“主线高亮 + 分支折叠”渲染
4. 用户手动展开分支，更新 `manuallyExpandedNodeIds`
5. 点击批量打印 -> 聚合节点 -> 拉取附件 -> 去重 -> 预检 -> 打印

## 5. 验收标准

- [ ] 首屏自动展示固定付款主线
- [ ] 缺环自动跳过且标红，不中断追溯
- [ ] 手动展开仅一层，且不影响现有抽屉预览
- [ ] 批量打印范围严格等于“主线 + 手动展开”
- [ ] 打印前预检可见并准确
- [ ] 主线识别异常时可自动回退原逻辑

## 6. 实施分解

1. 主线识别器（纯前端派生，最小侵入）
2. 缺口虚拟节点渲染
3. 手动展开状态与主线状态分离
4. 批量打印聚合与去重
5. Mock 场景补齐（完整链路/缺付款申请/缺回单）
6. 回归验证（查询、节点点击、抽屉、重置、打印）

## 7. 风险与控制

- 风险：不同业务类型关系方向不一致导致主线识别误判  
  控制：增加“固定主线映射表 + 关系优先级 + 回退策略”。
- 风险：打印性能波动  
  控制：先做附件元数据聚合预检，分页拉取并发控制。
