## 📊 集团型架构设计（PRD v1.0）

### 核心设计原则

根据 PRD v1.0 Section 1.1：

```
法人 (Entity) ──┐
                ├──> 全宗1 (Fonds 1) ──> 档案数据
                ├──> 全宗2 (Fonds 2) ──> 档案数据
                └──> 全宗3 (Fonds 3) ──> 档案数据
```

**关键点**：
- ✅ **数据隔离键**: `fonds_no`（全宗号）- 这是**硬约束**
- ⚠️ **管理维度**: `entity_id`（法人ID）- 仅用于**治理、统计、合规台账**，**不作为数据隔离键**
- ✅ **一个法人可以对应多个全宗**

### 为什么这样设计？

1. **数据隔离更灵活**: 一个法人可能有多个业务单元，每个业务单元需要一个独立的全宗
2. **合规要求**: 档案管理规范要求以"全宗"为基本单位
3. **权限控制**: 基于全宗进行数据隔离，更符合档案管理实践

## ✅ 已实现的功能

### 1. 全宗隔离机制（后端）✅

**实现位置**:
- `FondsIsolationInterceptor` - MyBatis 拦截器，强制过滤 `fonds_no`
- 所有业务表都包含 `fonds_no` 字段
- 从登录态解析 `allowed_fonds`，不信任请求入参

**效果**: 
- ✅ 用户只能看到自己授权范围内的全宗数据
- ✅ 跨全宗访问需要授权票据

### 2. 全宗管理（前端+后端）✅

**后端**:
- ✅ `BasFondsController` - 全宗 CRUD API
- ✅ `BasFondsService` - 全宗业务逻辑

**前端**:
- ✅ `FondsManagement.tsx` - 全宗管理页面
- ✅ 全宗列表、创建、编辑、删除

**访问路径**: `/system/admin` → 全宗管理标签

### 3. 全宗沿革管理（前端+后端）✅

**功能**: 全宗迁移、合并、分立、重命名

**访问路径**: `/system/admin/fonds-history`

### 4. 跨全宗访问授权（前端+后端）✅

**功能**: 申请、审批跨全宗访问授权票据

**访问路径**: `/system/security/auth-ticket`

### 5. 全宗切换器（前端）✅

**功能**: 在多个全宗间切换

**位置**: 顶部导航栏（TopBar）

**需要实现**:

```java
// EntityController.java
@RestController
@RequestMapping("/api/entity")
public class EntityController {

```

**数据库表**:
```sql
CREATE TABLE sys_entity (
    id VARCHAR(64) PRIMARY KEY,
    name VARCHAR(255) NOT NULL COMMENT '法人名称',
    tax_id VARCHAR(50) COMMENT '税号',
    address VARCHAR(500) COMMENT '地址',
    contact_person VARCHAR(100) COMMENT '联系人',
    contact_phone VARCHAR(50) COMMENT '联系电话',
    status VARCHAR(20) DEFAULT 'ACTIVE' COMMENT '状态',
    created_time TIMESTAMP,
    updated_time TIMESTAMP
);
```
