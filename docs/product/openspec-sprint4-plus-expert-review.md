## 一、法律法规与会计规范对标（合规专家审查）

### 1.1 电子会计档案销毁合规性审查

#### ✅ 合规点

1. **保管期限到期识别**: OpenSpec 4.2.1 明确要求基于 `doc_date + retention_policy.years` 计算到期，符合《会计档案管理办法》关于保管期限的要求。
2. **销毁审批流程**: 4.2.3 要求双人审批（初审+复核），符合档案销毁审批的合规要求。
3. **销毁清册永久保留**: 4.2.6 要求销毁清册永久只读，符合审计追溯要求。

#### ⚠️ 高风险问题

**问题 1: 销毁前鉴定清单的法规依据不明确**

- **关联规范**: 《会计档案管理办法》要求销毁前需进行鉴定，但 OpenSpec 未明确鉴定清单的具体内容要求。
- **风险**: 鉴定清单可能缺少法规要求的必备信息（如档案形成时间、保管期限起算日期、鉴定结论等）。
- **证据来源**: OpenSpec 4.2.2 仅要求"包含档案元数据快照"，未明确是否包含鉴定结论、鉴定人、鉴定日期等。
- **验证方式**: 对照《会计档案管理办法》关于销毁鉴定的具体条款（待核对）。

**建议**: 
- 补充鉴定清单的必备字段：鉴定人、鉴定日期、鉴定结论（同意销毁/不同意销毁/延期保管）。
- 鉴定清单应作为销毁审批的前置条件，而非可选步骤。

**问题 2: 销毁清册快照内容可能不完整**

- **关联规范**: 销毁清册需包含足以追溯档案原始信息的完整记录。
- **风险**: 当前快照 JSON 格式（4.2.6）未明确包含档案的**原始形成时间**、**归档时间**、**保管期限起算日期**等关键追溯信息。
- **证据来源**: OpenSpec 4.2.6 快照格式示例仅包含 `docDate`、`amount` 等业务字段，缺少档案管理维度的时间戳。
- **验证方式**: 对照 DA/T 94-2022 关于销毁清册的元数据要求（待核对）。

**建议**:
- 快照 JSON 应补充：`archivedAt`（归档时间）、`retentionStartDate`（保管期限起算日期）、`originalFormationDate`（原始形成时间）。

#### ⚠️ 中风险问题

**问题 3: 到期档案自动识别的时间计算逻辑需明确**

- **风险**: `doc_date + retention_policy.years` 的计算方式未明确是否考虑"保管期限起算日期"（可能不是 `doc_date`）。
- **证据来源**: OpenSpec 4.2.1 技术规格中仅提到 `doc_date`，但实际保管期限可能从归档日期或会计年度结束日期起算。
- **建议**: 明确保管期限起算规则，并在 `archive_object` 表中增加 `retention_start_date` 字段。

### 1.2 会计专业性合理性检查

#### ✅ 合理性

1. **在借校验**: 4.2.4 要求在借档案禁止销毁，符合档案管理实务。
2. **冻结/保全机制**: 4.2.7 支持审计/诉讼冻结，符合合规要求。

#### ⚠️ 中风险问题

**问题 4: 销毁执行的事务一致性保障不足**

- **风险**: 4.2.5 要求"物理文件删除前，必须确保销毁清册已写入"，但未明确事务回滚机制。若清册写入失败，物理文件可能已删除，导致数据丢失。
- **证据来源**: OpenSpec 4.2.5 关键实现点中仅提到"事务一致性"，但未说明具体的事务边界与回滚策略。
- **建议**: 
  - 明确事务边界：清册写入 → 元数据更新 → 物理文件删除（或标记删除）应在同一事务中。
  - 若物理文件删除失败，应回滚清册写入，避免数据不一致。

## 二、系统架构与安全深度审计（架构专家审查）

### 2.1 数据持久化与长期保存能力

#### ✅ 架构稳健性

1. **数据库表设计**: `destruction_log` 使用复合主键 `(id, fonds_no, archive_year)`，符合 PRD 要求。
2. **哈希链支持**: 4.2.6 要求 `prev_hash`/`curr_hash`，支持验真。

#### ⚠️ 高风险问题

**问题 5: 销毁清册表缺少唯一性约束与防重放机制**

- **风险**: `destruction_log` 表未明确唯一性约束（如 `UNIQUE(archive_object_id, fonds_no, archive_year)`），可能导致同一档案被重复记录销毁，或清册数据被篡改。
- **证据来源**: OpenSpec 4.2.6 数据库表结构未定义唯一性约束。
- **验证方式**: 检查 DDL 是否包含唯一索引或约束。
- **建议**: 
  - 增加唯一性约束：`UNIQUE(archive_object_id, fonds_no, archive_year)`。
  - 销毁执行前检查是否已存在清册记录，防止重复销毁。

**问题 6: 物理文件删除的不可逆风险**

- **风险**: 4.2.5 要求删除物理文件，但未明确删除前的备份策略。若删除后需要恢复，将无法追溯。
- **证据来源**: OpenSpec 4.2.5 关键实现点提到"支持软删除（标记删除）与硬删除（物理删除）两种模式（可配置）"，但未明确默认模式与备份要求。
- **建议**: 
  - **默认使用软删除**（标记删除），物理文件保留在隔离存储区。
  - 硬删除需额外审批，且删除前必须完成备份验证。
  - 删除操作需记录审计日志，包含文件路径、大小、哈希值。

#### ⚠️ 中风险问题

**问题 7: 定时任务扫描的性能与并发安全**

- **风险**: 4.2.1 定时任务每日扫描全表，若数据量大（百万级），可能导致长时间锁表或性能问题。
- **证据来源**: OpenSpec 4.2.1 未明确扫描策略（分页/分批）与并发控制。
- **建议**: 
  - 使用分页扫描，每批处理 1000 条。
  - 添加分布式锁，防止多实例重复执行。
  - 记录扫描进度，支持断点续扫。

### 2.2 电子会计档案"四性"技术实现

#### ✅ 四性保障

1. **完整性**: 销毁清册哈希链支持验真。
2. **安全性**: 冻结/保全机制防止误销毁。

#### ⚠️ 高风险问题

**问题 8: 销毁清册的不可篡改性保障不足**

- **风险**: 虽然要求哈希链，但未明确如何防止数据库管理员直接修改清册记录。
- **证据来源**: OpenSpec 4.2.6 要求"永久只读"，但未说明数据库层面的实现方式（如触发器、只读表、WORM 存储）。
- **建议**: 
  - 使用数据库触发器禁止 UPDATE/DELETE 操作（PostgreSQL/达梦支持）。
  - 或使用只读表空间（WORM 模式）。
  - 定期导出清册到独立存储（如对象存储的不可变桶）。

### 2.3 权限、审计与三员分立

#### ✅ 权限设计

1. **审批流程**: 双人审批符合权限分离要求。

  - 执行操作需记录审计日志，包含执行人、执行时间、TraceID。

### 合规专家建议

1. **补充鉴定清单的法规依据**: 明确鉴定清单的必备字段（鉴定人、鉴定日期、鉴定结论），确保符合《会计档案管理办法》要求。
