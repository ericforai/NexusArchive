# 数据库设计说明

本文档详细介绍 NexusArchive 系统的数据库架构设计。

---

## 设计理念

电子会计档案系统的数据库本质是 **"法律证据库"**：

> **合规性 > 性能**

---

## 核心规范

### 金额字段
- **类型**: 必须使用 `DECIMAL(18, 2)`
- **禁止**: 使用 `FLOAT/DOUBLE`
- **Java 端**: 使用 `java.math.BigDecimal`

### 哈希算法
- **默认**: SM3 (国密)
- **降级**: 如环境不支持自动降级为 SHA-256

### 文件存储
- 数据库仅存储元数据 (`arc_file_content`)
- 实体文件存储于 OSS 或 NAS

---

## 核心表结构

系统基于 **DA/T 94-2022** 标准构建了四大核心支柱表：

### 1. 单据总表 (`acc_archive`) - "户口本"

| 字段 | 说明 |
|------|------|
| `unique_biz_id` | 唯一业务ID，解决多源异构关联 |
| `amount` | 金额 |
| `doc_date` | 业务日期 |
| `volume_id` | 所属案卷 |
| `metadata_json` | 扩展元数据（NoSQL设计） |

### 2. 文件实体表 (`arc_file_content`) - "金库"

| 字段 | 说明 |
|------|------|
| `original_hash` | 接收时计算的哈希值 |
| `current_hash` | 巡检时计算的哈希值 |
| `timestamp_token` | 时间戳令牌 |
| `sign_value` | 电子签名值 |

**双哈希机制**：通过比对 `original_hash` 和 `current_hash` 实现自动健康巡检。

### 3. 案卷表 (`acc_archive_volume`) - "文件夹"

| 字段 | 说明 |
|------|------|
| `volume_code` | 案卷号 |
| `retention_period` | 保管期限 |
| `status` | 封卷状态 |
| `validation_report_path` | 四性检测报告路径 |

### 4. 审计日志表 (`sys_audit_log`) - "黑匣子"

| 字段 | 说明 |
|------|------|
| `data_before` | 变更前数据快照 |
| `data_after` | 变更后数据快照 |
| `ip_address` | 操作IP |
| `mac_address` | 操作MAC地址 |

**设计原则**：只增不改 (Append Only)，用于证据保全。

### 5. 符合性检查扩展

`audit_inspection_log` 表新增符合性检查字段：
- `is_compliant` - 是否符合
- `compliance_violations` - 违规项
- `compliance_warnings` - 警告项

---

## 四性检测逻辑

系统内置的健康巡检服务 (`ArchiveHealthCheckService`) 对已归档数据执行以下检查：

### 1. 真实性 (Authenticity)

- **逻辑**: 读取磁盘物理文件 → 重新计算哈希值 (SM3/SHA256) → 与数据库 `original_hash` 比对
- **作用**: 确保文件未被篡改，且磁盘存储完好

### 2. 完整性 (Integrity)

- **逻辑**: 校验元数据中关键合规字段 (`unique_biz_id`, `amount`, `doc_date`) 是否缺失
- **作用**: 确保档案元数据符合 DA/T 94-2022 标准

### 3. 可用性 (Usability)

- **逻辑**: 检查物理文件是否存在、是否可读；校验文件头 (Magic Number) 是否与扩展名匹配
- **作用**: 防止文件损坏或格式错误导致无法打开

### 4. 安全性 (Safety)

- **逻辑**: 调用病毒扫描接口对文件内容进行扫描
- **作用**: 确保存储环境安全，防止恶意软件潜伏

## 权限相关表

- `sys_org`：组织/部门
- `sys_role`：角色
- `sys_user`：用户
- `sys_user_role`：用户-角色关联
- `sys_permission`：权限配置
- `sys_setting`：系统配置
