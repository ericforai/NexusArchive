# 签章和时间戳功能实现分析报告

## 📋 问题分析

### 发现的问题

1. **签章服务已实现，但缺少控制器暴露API**
   - ✅ 服务层完整：`Sm2SignatureService`, `DigitalSignatureService`, `SignatureAdapter`
   - ✅ 数据层完整：`ArcSignatureLog`, `ArcSignatureLogMapper`
   - ❌ **缺少控制器**：没有 `SignatureController` 暴露签章API

2. **时间戳服务完全缺失**
   - ❌ 没有时间戳服务实现
   - ❌ 没有时间戳控制器
   - ❌ 没有时间戳配置

3. **证书管理功能缺失**
   - ❌ 没有证书管理控制器
   - ✅ 密钥库加载逻辑在 `Sm2SignatureService` 中，但未暴露API

## ✅ 已实现的解决方案

### 1. 创建了签章控制器 (`SignatureController`)

**文件**: `nexusarchive-java/src/main/java/com/nexusarchive/controller/SignatureController.java`

**功能**:
- ✅ `POST /api/signature/sign` - 对数据进行签名
- ✅ `POST /api/signature/verify` - 验证签名
- ✅ `POST /api/signature/verify-pdf` - 验证PDF文件签章
- ✅ `POST /api/signature/verify-ofd` - 验证OFD文件签章
- ✅ `GET /api/signature/logs` - 查询签章日志
- ✅ `GET /api/signature/status` - 检查签章服务状态

**特点**:
- 自动记录签章日志到 `arc_signature_log` 表
- 支持关联档案ID和文件ID
- 完整的错误处理

### 2. 创建了时间戳服务 (`TimestampService`)

**文件**: `nexusarchive-java/src/main/java/com/nexusarchive/service/TimestampService.java`

**功能**:
- ✅ 支持 RFC 3161 Time-Stamp Protocol (TSP)
- ✅ 支持超时降级（使用本地时间）
- ✅ 支持时间戳验证
- ✅ 可配置的TSA URL、用户名、密码
- ✅ 可配置的超时时间

**特点**:
- 当TSA服务不可用时，自动降级使用本地时间
- 完整的错误处理和日志记录
- 支持基本认证

### 3. 创建了时间戳控制器 (`TimestampController`)

**文件**: `nexusarchive-java/src/main/java/com/nexusarchive/controller/TimestampController.java`

**功能**:
- ✅ `POST /api/timestamp/request` - 请求时间戳
- ✅ `POST /api/timestamp/verify` - 验证时间戳
- ✅ `GET /api/timestamp/status` - 检查时间戳服务状态

### 4. 创建了证书管理控制器 (`CertificateController`)

**文件**: `nexusarchive-java/src/main/java/com/nexusarchive/controller/CertificateController.java`

**功能**:
- ✅ `GET /api/admin/certificates` - 查询所有证书
- ✅ `GET /api/admin/certificates/{alias}` - 查询证书详情
- ✅ `POST /api/admin/certificates/{alias}/verify` - 验证证书有效性

**特点**:
- 自动检查证书是否过期
- 显示证书详细信息（主题、颁发者、有效期等）
- 支持证书验证

### 5. 更新了测试脚本

## 🔧 配置要求

### 1. 签章服务配置

在 `application.yml` 中添加：

```yaml
signature:
  keystore:
    path: classpath:keystore/signature.p12  # 或绝对路径
    password: changeit
```

### 2. 时间戳服务配置

在 `application.yml` 中添加：

```yaml
timestamp:
  enabled: true
  tsa:
    url: http://tsa.example.com/tsp
    username: tsa_user  # 可选
    password: tsa_password  # 可选
    timeout: 5000
  fallback-on-error: true  # 错误时使用本地时间
```

## 🚀 使用流程

### 签章流程

1. **检查签章服务状态**
   ```bash
   GET /api/signature/status
   ```

2. **查询可用证书**
   ```bash
   GET /api/admin/certificates
   ```

3. **对数据进行签名**
   ```bash
   POST /api/signature/sign
   {
     "data": "base64_encoded_data",
     "certAlias": "signing-cert",
     "archiveId": "ARCHIVE-001"
   }
   ```

4. **验证签名**
   ```bash
   POST /api/signature/verify
   {
     "data": "base64_encoded_data",
     "signature": "base64_encoded_signature",
     "certAlias": "signing-cert"
   }
   ```

### 时间戳流程

1. **检查时间戳服务状态**
   ```bash
   GET /api/timestamp/status
   ```

2. **请求时间戳**
   ```bash
   POST /api/timestamp/request
   {
     "data": "base64_encoded_data",
     "archiveId": "ARCHIVE-001"
   }
   ```

3. **验证时间戳**
   ```bash
   POST /api/timestamp/verify
   {
     "data": "base64_encoded_data",
     "timestampToken": "base64_encoded_token"
   }
   ```

## ⚠️ 注意事项

### 1. 密钥库配置

- 确保密钥库文件存在且可访问
- 密钥库格式必须是 PKCS12 (.p12)
- 确保密码正确

### 2. TSA 服务配置

- TSA URL 必须可访问
- 如果 TSA 不可用，系统会降级使用本地时间（如果启用）
- 建议在生产环境使用可信的 TSA 服务

### 3. 证书管理

- 证书存储在密钥库中
- 定期检查证书有效期
- 过期证书会导致签章失败

## 🔍 故障排除

### 签章服务不可用

**可能原因**:
1. 密钥库路径不正确
2. 密钥库密码错误
3. BouncyCastle 提供者未注册
4. 证书不存在或已过期

**解决方法**:
1. 检查 `signature.keystore.path` 配置
2. 检查 `signature.keystore.password` 配置
3. 查看日志中的错误信息
4. 使用 `/api/signature/status` 检查服务状态

### 时间戳服务不可用

**可能原因**:
1. TSA URL 不可访问
2. 网络连接问题
3. 超时设置过短
4. 认证信息错误

**解决方法**:
1. 检查 `timestamp.tsa.url` 配置
2. 检查网络连接
3. 增加 `timestamp.tsa.timeout` 值
4. 检查认证信息
5. 如果启用降级，系统会使用本地时间
