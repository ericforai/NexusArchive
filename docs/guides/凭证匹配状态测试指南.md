我已完成凭证匹配状态流的深度优化，涵盖了从后端枚举定义到前端视觉反馈的全链路。以下是您的手工测试步骤：

### 1. 测试准备
确保后端服务已根据我的最新改动成功启动。我已经通过 SQL 手工将测试数据 `YS-2025-10-记-99` 的状态设为“匹配成功”：
- **状态 (Status)**: `MATCHED`
- **匹配得分 (Score)**: `100`
- **关联方式 (Method)**: `金额+日期精确匹配`

### 2. 前端验证步骤
1. **访问系统**: 打开浏览器，进入 [http://localhost:5175](http://localhost:5175)。
2. **进入菜单**: 
   - 导航至 **“档案管理”** -> **“会计凭证”**（Repository 视图）。
   - 或者导航至 **“预归档库”** -> **“凭证关联”**（Smart Matching 视图）。
3. **观察状态标签**:
   - 找到凭证 `YS-2025-10-记-99`。
   - **状态列**: 应显示为翠绿色的 **“匹配成功”** 标签。
4. **观察视觉标记**:
   - 在 **凭证号** 旁边，您应该能看到一个带有“闪电”图标的 **“已匹配”** 小勋章。
   - 这是本次新增的视觉特性，用于快速识别高信心关联。
5. **如果是“凭证关联”视图**:
   - 您还会在列表中查看到 **“匹配度”**（100%）和 **“关联方式”**（金额+日期精确匹配）。

### 3. 技术变更清单 (供参考)
- **后端枚举**: `PreArchiveStatus` 新增 `DRAFT` (草稿), `MATCH_PENDING` (待匹配), `MATCHED` (匹配成功)。
- **数据库扩展**: `acc_archive` 增加了 `match_score` 和 `match_method` 字段。
- **自动逻辑**: 修改了 `VoucherMatchingEngine`，匹配完成后会自动回写这些分值和状态。
- **导入优化**: 修改了 `ProcessingListener`，新导入的凭证初始状态现在是 `MATCH_PENDING`。
