## 二、角色体系

### 2.1 角色分类

#### 管理员角色（三员分立）

符合 **GB/T 39784-2021**《电子档案管理系统通用要求》和等保 2.0 三级标准：

| 角色 | 代码 | 职责范围 | 数据范围 | 互斥 |
|------|------|----------|----------|------|
| **系统管理员** | `system_admin` | 系统运维、配置管理、备份恢复、性能监控 | `all` | ✅ |
| **安全保密员** | `security_admin` | 用户管理、角色管理、权限分配、组织架构管理 | `all` | ✅ |
| **安全审计员** | `audit_admin` | 审计日志查看、审计报表导出、操作追踪 | `all` | ✅ |

**互斥规则**：
- 三员角色不能同时分配给同一用户（前后端双重校验）
- `is_exclusive = true` 标识互斥角色
- 分配时自动检测并拒绝冲突分配

#### 业务角色

| 角色 | 代码 | 描述 | 权限范围 |
|------|------|------|----------|
| **超级管理员** | `super_admin` | 拥有所有权限，仅用于开发/测试环境 | `nav:all` + 全部操作权限 |
| **业务操作员** | `business_user` | 默认业务用户，具有基本业务操作权限 | 全部导航菜单，无管理权限 |
| **查询用户** | `query_user` | 仅能通过借阅审批访问特定档案 | 最小权限 |

## 三、全宗权限机制

### 3.1 全宗隔离原理

全宗是档案管理的基本单位，每个法人实体对应一个全宗。系统强制实施全宗隔离：

1. **任何用户必须分配至少一个全宗**，否则无法访问系统
2. **超级管理员**分配全宗后可跨全宗查询所有数据
3. **普通用户**只能访问分配的全宗数据

### 3.2 全宗上下文 (FondsContext)

**后端实现**：`FondsContextFilter.java`

```java
// 请求头设置当前全宗
X-Fonds-Code: F001

// 过滤器验证流程
1. 检查用户是否登录
2. 检查用户是否分配了全宗
3. 验证请求的全宗是否在用户允许列表中
4. 设置 FondsContext 当前全宗
5. 业务层通过 FondsContext.get() 获取当前全宗
```

**前端实现**：全宗切换器

```
┌─────────────────────────────────┐
│  当前全宗: [测试公司全宗 ▼]     │
├─────────────────────────────────┤
│  • 测试公司全宗                 │
│  • 演示公司全宗                 │
└─────────────────────────────────┘
```

### 3.3 用户全宗权限表

`sys_user_fonds_scope` 结构：

## 四、权限检查机制

### 4.1 后端权限检查

#### JWT 认证流程

```
┌─────────────┐     JWT Token      ┌──────────────────┐
│   前端请求   │ ──────────────────> │ JwtAuthFilter   │
│  + Header    │                    │  1. 验证Token    │
│  Authorization│                    │  2. 加载用户信息 │
└─────────────┘                    │  3. 加载角色权限 │
                                   │  4. 加载全宗列表 │
                                   └────────┬─────────┘
                                            │
                                            ▼
                                   ┌──────────────────┐
                                   │ FondsContextFilter│
                                   │  1. 验证全宗权限  │
                                   │  2. 设置当前全宗  │
                                   └────────┬─────────┘
                                            │
                                            ▼
                                   ┌──────────────────┐
                                   │   Controller    │
                                   │   @PreAuthorize │
                                   └─────────────────┘
```

#### 权限注解

```java
@PreAuthorize("hasAuthority('archive:delete')")
public Result<Void> deleteArchive(@PathVariable String id) {
    // 只有具有 archive:delete 权限的用户才能调用
}
```

#### 特殊拦截器

**BorrowPermissionInterceptor** - 查询用户专用拦截器

```java
// 拦截规则
if (isQueryUser()) {
    // 禁止下载
    if (isDownloadRequest(uri)) {
        return false; // "查询用户无权下载档案，仅支持在线预览"
    }
    
    // 禁止列表访问
    if (isArchiveListRequest(uri)) {
        return false; // "请通过借阅申请流程访问档案"
    }
    
    // 验证借阅权限
    if (!borrowingFacade.checkAccess(userId, archiveId, "VIEW")) {
        return false; // "无权访问该档案，请先提交借阅申请"
    }
}
```

### 4.2 前端权限检查

#### useAuthStore 状态管理

```typescript
interface AuthState {
  user: User | null;
  permissions: string[];
  roles: string[];
  
  // 权限检查方法
  hasPermission: (perm: string) => boolean;
  hasAnyPermission: (perms: string[]) => boolean;
  hasRole: (role: string) => boolean;
  isSuperAdmin: () => boolean;
}
```

#### 递归菜单权限过滤

**Sidebar.tsx** 中的 `filterNavItems()` 函数：

```typescript
const filterNavItems = (items: NavItem[]): NavItem[] => {
  return items.filter(item => {
    // 检查当前项权限
    if (item.permission && !hasPermission(item.permission)) {
      return false;
    }
    
    // 递归过滤子项
    if (item.children?.length > 0) {
      const filteredChildren = filterNavItems(item.children);
      item.children = filteredChildren;
      // 如果所有子项都被过滤掉，则隐藏父项
      return filteredChildren.length > 0;
    }
    
    return true;
  });
};
```

#### 菜单权限配置示例


## 五、安全特性

### 5.1 密码策略

- **长度要求**：≥ 8 位
- **复杂度要求**：必须包含大小写字母、数字、特殊字符
- **登录失败锁定**：5 次失败后锁定 15 分钟（Redis 分布式锁定）

### 5.2 Token 安全

- **JWT 签名**：使用 SHA256 哈希签名
- **Token 黑名单**：登出后 Token 加入黑名单
- **自动刷新**：Token 过期前自动刷新

### 5.3 审计日志

- **防篡改机制**：SM3 哈希链
- **日志内容**：操作前数据、操作后数据、IP 地址、操作时间
- **哈希验证**：每条日志记录 `log_hash` 和 `prev_log_hash`

```java
log_hash = SM3(content + prev_log_hash)
```

### 5.4 数据加密

| 算法 | 用途 |
|------|------|
| **SM3** | 审计日志哈希链、文件完整性校验 |
| **SM4** | 敏感字段加密 |
| **SM2** | 电子签章验签 |
| **Argon2** | 密码哈希存储 |

## 八、故障排查

### 8.1 用户无法登录

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 提示"用户不存在" | 用户已删除或输入错误 | 检查用户名是否正确 |
| 提示"密码错误" | 密码错误 | 重置密码 |
| 提示"账号已锁定" | 登录失败次数过多 | 等待15分钟或管理员解锁 |
| 提示"无全宗权限" | 未分配全宗 | 分配至少一个全宗 |

### 8.2 菜单显示不全

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 只能看到部分菜单 | 角色权限不足 | 联系管理员添加权限 |
| 子菜单全部隐藏 | 所有子菜单权限缺失 | 检查角色权限配置 |
| 退出登录后仍有菜单 | 浏览器缓存 | 清除缓存并重新登录 |

### 8.3 全宗切换失败

| 现象 | 可能原因 | 解决方案 |
|------|----------|----------|
| 无法切换全宗 | 未分配该全宗权限 | 联系管理员分配全宗 |
| 切换后无数据 | 全宗内无数据 | 确认全宗数据存在 |
